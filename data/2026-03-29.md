Why is intent engineering so important?  Case study: Klarna  Klarna deployed an AI agent that replaced humans and saved them $60 million a year.  & all the numbers looked flawless.  Month one: 2.3 million conversations handled across 23 markets and 35 languages.  Workload equivalent: 853 full-time employees.  Average ticket resolution: dropped from 11 minutes to 2 minutes.  Savings: $60 million.  On paper? A masterclass.  By mid-2025, Klarna was forced to rehire the human agents it had terminated.  The AI had been implicitly told to “resolve tickets fast.”  And its agent optimised for that metric ruthlessly.  Here’s what the AI couldn’t do:  A human agent intuitively knows that when a highly profitable 3-year customer is severely frustrated, the right move isn’t to ignore the resolution-time metric.  It’s to escalate the issue, prioritise empathy, and protect long-term retention.  That’s tacit organisational knowledge.  That’s intent.  The AI lacked the capacity to balance trade-offs. It optimised perfectly for exactly the wrong objective.  This is what we call the “intent gap.”  A technically brilliant agent optimising perfectly for the wrong thing.  Why? Because organisational purpose was never encoded into the infrastructure.  Klarna had world-class prompt engineering.  Klarna had solid context integration.  Klarna had zero intent engineering.  It saved $60 million but is the poster boy face of “AI gone wrong.”  If you want to know how to utilise:  > prompt engineering > context engineering > intent engineering   Then read the following article & learn how to become an AI engineer:

## 关键词 "LLM"

### 03:06 @unknown

Google DeepMind has unveiled a browser powered by its Gemini 3.1 Flash-Lite model that generates entire websites in real time as users browse.   Google’s Flash-Lite Browser treats the web like something an LLM can write live, not something humans must fully pre-build first.  A normal site serves stored pages and templates, but this system uses Gemini 3.1 Flash-Lite to generate fresh HTML and CSS from your prompt, clicks, and navigation context almost instantly.  The technical shift is simple: instead of fetching a finished page, the browser asks the model what page should exist right now, then streams that answer as interface code.  That makes personalization much deeper, because the page can change for each user, each step, and each goal without keeping a huge library of prewritten screens.  It also fits agentic workflows, where an AI assistant may need to create a temporary tool, dashboard, or reference page on the fly while working through a task.  IMO, the catch is reliability, because once page layout and content are model outputs, bugs, hallucinations, style drift, and serving cost - all become concerns.
