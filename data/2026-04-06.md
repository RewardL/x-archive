The best AI teams are not winning on models. They are winning on harness engineering. And almost nobody is talking about it.  I spent a week writing 8,000 words on harness engineering.  Here's what actually matters, in 8 points:  >The interface is not decoration. For an LLM agent, the interface is the mind.  >The context window is not a RAM slot. Every irrelevant token you dump into it competes with the information that actually matters, and the model does not cleanly ignore the noise.  >Capping search results outperforms returning everything. When you force the agent to refine its query instead of flooding it with matches, you turn a failure mode into a feedback loop.  >Run a linter at edit time, not at test time. A syntax error caught at the moment of writing costs nothing. The same error caught ten steps later cascades into a debugging spiral that burns your entire context budget.  >A persistent progress file is the cheapest fix in agentic systems that almost nobody uses. Continuity across context window boundaries is not a hard problem. It is a text file the agent reads at start and writes at the end.  >Make completion criteria explicit and binary. If the agent has to read the code and guess whether a feature is done, it will guess wrong often enough to matter.  >One agent, one git worktree. Isolation is not overhead. It is how you run agents in parallel without them stepping on each other.  >Your agent's ceiling is your feedback loop. If it cannot see the consequences of what it built, it cannot fix what actually broke.  Read the article, if you want to know in detail.

🚨 Andrej Karpathy just dropped something that could replace a lot of RAG workflows.  It’s called LLM Wiki.  The idea is simple:  Most AI systems retrieve context from scratch every time you ask a question.  LLM Wiki doesn’t.  It builds a persistent knowledge base that gets better every time you add a new source.  So instead of:  • search docs • pull fragments • answer • forget everything • repeat  it does this:  • ingest a source • extract the important ideas • update entity pages • revise topic summaries • connect related concepts • flag contradictions • keep compounding the knowledge over time  That shift matters.  RAG is great for retrieval.  But a lot of people are really trying to build memory.  Not just “find me the right chunk again.” More like: “help me build an evolving model of this topic over time.”  That’s what this is.  Karpathy’s examples are strong too:  • personal knowledge • long-horizon research • books and topics • internal company knowledge • meeting transcripts • customer calls  Basically, anything where the knowledge should accumulate, not reset every session.  The best way to think about it:  Obsidian is the IDE. The LLM is the programmer. The wiki is the codebase.  You don’t manually maintain the system.  You feed it sources, ask questions, and the AI keeps the structure alive.  That’s a much bigger idea than “better RAG.”  100% open source.
